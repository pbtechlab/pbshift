# pbPitchShift (`pbshift`)

<img src="docs/assets/banner.svg" alt="pbshift — リアルタイム・ピッチシフト／タイムストレッチ" width="100%">

**[English README (default) → README.md](README.md)**

リアルタイム**ピッチシフト／タイムストレッチ C++ ライブラリ**。倍音の純度・トランジェントの鋭さ・フォルマントの自然さ・ビット完全な再現性が、互いを犠牲にすることなく同時に成り立つように設計・実装されています。

- **C++17 静的ライブラリ**。外部ランタイム依存ゼロ（同梱する BSD ライクな FFT が 1 つだけ）
- **ストリーミング・プル型 API**（`feed` → `available` → `read`、`finish` でテールをフラッシュ）。リアルタイムインサートにもオフラインレンダリングにも同じコードで対応
- **ピッチ ±24 半音**、**タイムストレッチ 0.25×–4×**（範囲外の値はエンジン内部でクランプ）。両者は独立で、同時指定も可能
- **フォルマント保持**: True Envelope クラスのスペクトル包絡エンジン
- **マルチ解像度タイムストレッチエンジン**（オフライン）— 帯域ごとに窓サイズを変え、内容適応でレイアウトを自動選択。密なミックス・パッド・持続母音で発生するフェーズボコーダ特有の「コーラス感／水っぽい」アーティファクトを、トランジェントを鋭く保ったまま除去
- **決定論的**: 同じ入力＋同じ設定 ⇒ ホストのバッファサイズに依存せず、再レンダリングでも**ビット完全一致**の出力

> ステータス: 活発に開発中（pre-1.0）。以下に示すストリーミング API が安定した公開サーフェスです。

## pbshift の狙い

タイムスケール変換は本質的にトレードオフを強います。倍音の純度**か**、トランジェントの鋭さ**か**、フォルマントの自然さ**か**、再現可能なレンダリング**か**。pbshift はこれらが互いを劣化させないように設計されています。

- 単一の共有解析フロントエンドがすべてのサブエンジンに情報を供給します。ゲインや位相がずれていく並列処理チェーンは存在しません。
- トランジェントはスペクトルパイプラインで塗り広げられるのではなく、スパースな時間領域イベントとして扱われます。
- すべての確率的要素は**ストリーム位置でシード**されるため、ランダム性が再現性を壊すことはありません。
- バイパス経路（ストレッチ 1.0、ピッチ 0）は最初のコミットから厳格なヌルテストゲートで管理されており、エンジンが信号を勝手に着色することはできません。

## 機能詳細

### ピッチシフト

- 範囲: **−24 〜 +24 半音**（連続値、小数も可）。
- エンジン内蔵の Kaiser 窓 sinc リサンプラとスペクトルストレッチエンジンの組み合わせで実装。
- **方向適応的なリサンプラ配置**: 下方向シフトではリサンプラをエンジンの**前段**に（アーティキュレーションを引き伸ばしてから解析へ）、上方向シフトでは**後段**に配置し（アーティキュレーションの時間圧縮を回避）、シフト方向ごとに自動選択します。この配置により子音やアタックの自然さが計測可能なレベルで向上します。

### タイムストレッチ

- 範囲: **0.25× 〜 4×**（比率 = 出力時間 / 入力時間）。範囲外の値も受け付け、エンジンがクランプします。
- ピッチとストレッチは完全に独立しており、同時指定できます（例: +7 半音 × 1.5 倍）。
- 深い圧縮時には解析ホップを自動的に短縮し、振幅の時間変化がエイリアスしない（フレーム間でコンテンツが飛ばされない）ようにします。

### マルチモード（汎用・トランジェント適応）

`Music` / `Rhythm` モード（CLI `--multi`）は、ドラム・ミックス・その他の広帯域/打楽器素材向けにチューニングした、ストリーミングの汎用経路です。レートのピン留めやオンセット位相リセットといったスケジューリング的トリック（伸長したドラムにフラム/エコーを残す原因）を使わず、**鋭く単一（二重化しない）なトランジェント**と**Chorus のない持続音**を一つの機構群で両立します。

- **長い解析窓＋常時短い合成窓。** 長窓で周波数分解能を確保し、短い合成窓（デュアル窓 WOLA、Portnoff 1976 / Allen & Rabiner 1977）で、位相変更されたトランジェントをプリ/ポストエコーに広げる iFFT の裾を捨てます。デュアル `sig`/`norm` 再構成はどの窓の組でも厳密です。
- **全フレームでスペクトル全域の恒等位相ロック・コヒーレンス**（Laroche & Dolson 1999）を、解析前段の再割当**群遅延**で駆動。谷〜谷の各パーシャル区間を、そのピークの時間伝播位相を軸に剛体として再構築するため、重なるグレインがコム化せず単一インパルスに補強されます。
- **エネルギー急増パーシャルの併合**（トランジェント処理、Röbel 2003 参照）: 広帯域の打撃は多数のビンを立ち上げますが、前フレーム比でエネルギーが急増したパーシャルを少数の最強コヒーレント中心へ併合するため、打撃全体が少数の位相中心で支配され、デチューンした複製の集合ではなく一つの鋭いアタックになります。
- **途切れない位相連続性**（オンセットでリセットしない）により、重なる全グレインがトランジェントを一瞬に配置します。ステレオは既定経路と同じ ch 間位相の verbatim コピーで像を保持します。

オプトイン（`--multi` / `Config::Mode::Music` / `Rhythm`）。`Auto` と `Voice` はバイト単位で不変です。

### マルチ解像度タイムストレッチ（オフライン）

<img src="docs/assets/multires.svg" alt="マルチ解像度: 低域は長窓で周波数分解能（コーラス感除去）、高域は短窓で時間分解能（トランジェント維持）。レイアウトは内容適応" width="100%">

最も難しいタイムストレッチ素材 — 密なミックス、持続パッド、伸ばした母音 — は単一解析窓の本質的な限界を露呈させます。**短い**窓はトランジェントを鋭く保ちますが、近接する倍音を分離するだけの周波数分解能がなく、それらの位相がオーバーラップ加算グリッドとビートして、フェーズボコーダ特有の**「コーラス感／水っぽい」**アーティファクトを生みます。**長い**窓は倍音をきれいに分離しますが、アタックを滲ませます。オプションのマルチ解像度エンジンは、ハイエンドのストレッチャと同様に、**周波数帯域ごとに専用サイズの窓で処理する**ことでこれを解決します：

- **低域・長窓** — 高い周波数分解能で密な倍音をきれいに分離。これが持続音・複雑音のコーラス感／フラッターを除去する要です。
- **高域・短窓** — 高い時間分解能でアタックを鋭く保ちます。
- 帯域分割は**同一フレームタイミングの STFT 振幅領域**で行うため、境界をまたいで位相が整合したままです（時間領域クロスオーバーの「フラム」が出ません）。

帯域分割に加えて：

- **内容適応レイアウト。** 信号（スペクトル平坦度＋ピッチ周期性）を解析し、レイアウトを自動選択します：
  - 広帯域の**打楽器** → クロスオーバー無しの単一短窓（孤立クリックを鋭く保つ。帯域分割はクリックの低域エネルギーを長窓に広げてアタックを鈍らせる）；
  - 純粋な**持続トーン／伸ばした母音** → 単一長窓（全倍音が位相コヒーレントに保たれる。密な声の倍音をクロスオーバーで分断すると非相関化する）；
  - **それ以外**（トーナルなミックス、ポリフォニー、音声）→ マルチ解像度分割。
- **比率適応の窓・オーバーラップ。** 窓長と合成オーバーラップをストレッチ比率に適応させます — 伸長時は長窓（ゆっくり変化する倍音を追従）、圧縮時は高オーバーラップ（解析ホップを細かく保つ）— 0.25×–4× の全域で品質を維持します。
- **トランジェント・ピニング。** *広帯域*オンセット検出器がアタック区間で局所比率を 1.0 に固定し、時間的に引き伸ばさず、生じたタイミング債務を続く約 100 ms で返済します。任意のスペクトルフラックスではなく広帯域の立ち上がりに限定するため、トーナルなビブラート等で誤発火して振幅変調を注入することがありません。
- **ステレオ位相ロック。** 参照チャンネルが全位相／オンセット判定を駆動し、チャンネル間位相差をそのままコピーしてステレオ像を保持します。ストリーミング経路と同様に決定論的でビット完全一致です。

pbshift 自身の客観コーラス感指標（持続倍音上の 1–25 Hz 振幅変調深度＋トランジェントのアタック鋭さクレスト）では、マルチ解像度エンジンは信号クラス × 比率の組み合わせの大多数でコーラス感の測定値を下げ、特に中〜大ストレッチ（2×–4×）で最大の低減 — まさにコーラス感が最も可聴になる領域 — を示し、持続トーナル・音声・純音素材では全比率でノイズフロア近傍に保ちます。`benchmarks/ratio_eval.py` で再現できます。

**ステータス: オプトインのオフライン経路。** ストリーミングの既定は不変です。マルチ解像度エンジンは全信号（バッファリング）経路で、`Offline` ティアで選択します（[CLI の使い方](#cli-の使い方)参照）。知覚変化は既定へ昇格する前に必ず耳（MUSHRA 方式の試聴）で検証するという本プロジェクトの方針に従い、明示的スイッチでゲートしています。フォルマント保持を要求した場合はストリーミングエンジンへ透過的にフォールバックします。

### フォルマント保持

- オプション機能（`setFormantPreserve(true)`）。ピッチシフトされた音声など、フォルマントが重要な素材向け。
- **True Envelope クラス**のケプストラム・スペクトル包絡推定（max フィルタによるサブサンプリング＋適応的反復精緻化）を用い、ビンごとのゲインクランプ付きスペクトル・プリワープとして適用します。
- 包絡次数は FFT サイズごとにチューニング済み。子音的な文脈ではトランジェントのピン留めを自動的に抑制するモードゲートを備え、音声品質が計測可能に向上しました。

### 処理モードとレイテンシティア

`Config` はモードヒント（`Auto` / `Voice` / `Rhythm` / `Music`）とレイテンシティア（`Live` / `StudioRT` / `Offline`）を公開しています。現在の基準ティアは `StudioRT` です。`Music` / `Rhythm` はストリーミングの**マルチモード**（汎用のトランジェント適応経路。[下記](#マルチモード汎用トランジェント適応)参照）を有効化します。低レイテンシの `Live` ティアはロードマップの進行中項目です（[ロードマップ](#ロードマップ)参照）。`Auto` と `Voice` はマルチモード経路によってバイト単位で不変です。

### チャンネル・サンプルレート・バッファ

- 任意のチャンネル数（`Config::channels`）。ステレオ素材には専用のコヒーレンス処理があります（後述）。
- サンプルレートは設定可能で、内部の窓長はサンプルレートから導出されます（テスト・ベンチマークの基準は 48 kHz）。
- オーディオ I/O は**プレーナ（非インターリーブ）float**: `float* const buffers[channel][frame]`。
- 各 `Stretcher` インスタンスは完全に独立しており、グローバル状態はありません。

## アルゴリズム概要

<img src="docs/assets/pipeline.svg" alt="信号経路: 入力 → ピッチリサンプラ → 解析フロントエンド（リアサインメント演算子）→ 位相＋トランジェントエンジン（identity-locked 領域, hold-fire-pin）→ WOLA 合成 → 出力。ステレオは参照チャンネル位相ロック、決定論的ビット完全出力" width="100%">

pbshift は、公開された研究と自前の計測のみに基づいて特許に配慮しつつゼロから実装したエンジンです。信号経路:

```
入力
  → [フォルマント・プリワープ（True Envelope クラス）、オプション]
  → [ピッチリサンプラ（Kaiser-sinc）— ピッチ下げ時はエンジン前段に配置]
  → 共有解析フロントエンド（1 ホップにつき 1 回の STFT）:
        振幅、位相、
        ビンごとの瞬時周波数     （リアサインメント演算子、時間微分窓）
        ビンごとの群遅延         （リアサインメント演算子、時間重み窓）
  → 単一の合成ループ:
        倍音領域       : リアサインメント推定値に対する位相勾配積分＋
                         スペクトル領域内の恒等位相ロック
                         （信頼度がしきい値未満のビンには
                         位置シードのランダム位相を付与）
        トランジェント : ホールド → 発火 → ピン留め エンジン（後述）
  → ステレオ: 位相決定は基準チャンネル上で実施。入力のチャンネル間
              位相差はそのままコピー
  → [ピッチリサンプラ — ピッチ上げ時はエンジン後段に配置]
  → WOLA（重み付きオーバーラップ加算）合成
  → 出力
```

### 解析フロントエンド: リアサインメント演算子

フレーム間の位相差分から周波数を推定する古典的なフェーズボコーダの手法（ノイズが多くホップ依存）ではなく、フロントエンドは**単一フレームからリアサインメント演算子を計算**します。補助的な窓付き変換から、ビンごとの瞬時周波数とビンごとの群遅延を求めます。検証済みの精度（下記ゲート参照）: スペクトルピーク近傍の周波数誤差 **0.0002 セント**、インパルスに対する群遅延誤差 **1 サンプル未満**。

### 位相エンジン: 恒等位相ロック領域合成

出力位相は、計測された位相勾配（時間方向は瞬時周波数、周波数方向は群遅延）を時間周波数平面上で積分して生成します。最も信頼できる（振幅の大きい）ビンから順に処理します。1 つのスペクトル領域内のビンは**入力とまったく同じ相互位相関係にロック**されたままです。垂直コヒーレンスが構造的に保存されるため、倍音純度 134 dB という結果につながっています。信頼度しきい値未満のビンには**決定論的な位置シード**のランダム位相が与えられます。

### トランジェントエンジン: ホールド → 発火 → ピン留め

アタックはスペクトルフラックスの広帯域立ち上がり検出器で検出します（不応期があるため単一イベントの二重トリガは起きません）。検出されたアタックは次のように処理されます。

1. **ホールド** — アタックが近づく間、ストレッチグリッド上に塗り広げる代わりにフレームを一時的に保持します。
2. **発火** — アタックを正しく写像された出力位置（`t_out = ratio · t_in`）に、**元の長さと減衰のまま**出力します。
3. **ピン留め** — イベント周辺の局所ストレッチ比を 1 に固定し、そこで生じたタイミング負債を後続フレームで滑らかに返済します。

イベントに消費された入力フレームは消費済みとしてマークされるため、**トランジェントの二重化は構造的に不可能**です。結果: パーカッションテストでのアタック保持比 1.06（1.00 が理想）。

### ステレオコヒーレンス

すべての位相決定は基準チャンネル上で一度だけ行い、入力のチャンネル間位相差を出力にそのままコピーします。計測されたステレオコヒーレンスドリフト: **0.000**。

### 構造としての決定論

アルゴリズムがランダム性を必要とするすべての箇所（しきい値未満の位相、ノイズ励振）で、時刻・ポインタ・呼び出しパターンではなく**ストリームの絶対位置からシード**した PRNG を使います。決定論的な内部スケジューリングと組み合わせることで、ホストがバッファをどう切ろうともビット完全一致の出力が得られます。[決定論の保証](#決定論の保証)を参照してください。

## 公開 API

公開サーフェスはヘッダ 1 つだけです: [`include/pbshift/pbshift.h`](include/pbshift/pbshift.h)

```cpp
namespace pbshift {

struct Config {
    int sampleRate = 48000;
    int channels   = 2;
    enum class Mode { Auto, Voice, Rhythm, Music };
    Mode mode = Mode::Auto;
    enum class Tier { Live, StudioRT, Offline };
    Tier tier = Tier::StudioRT;
};

class Stretcher {
public:
    void configure(const Config& cfg);
    void reset();

    void setTimeStretch(double ratio);        // 出力/入力の時間比, [0.25, 4]
    void setPitchSemitones(double semitones); // [-24, +24]
    void setFormantPreserve(bool enable);

    void feed(const float* const* in, int frames); // プレーナ（非インターリーブ）
    void finish();                                 // 入力終了。テールをフラッシュ
    int  available() const;                        // 確定済み出力フレーム数
    int  read(float* const* out, int frames);      // 書き込んだフレーム数を返す

    int  inputLatency() const;   // サンプル数。ホストの PDC 用
    int  outputLatency() const;
};

} // namespace pbshift
```

### ストリーミングモデル

`Stretcher` は**プル型ストリーミングプロセッサ**です。

1. 任意のフレーム数を `feed()` します（チャンクサイズは自由 — 1 サンプルでも 100 万サンプルでも出力ビットは同一）。
2. `available()` が**確定済み**の出力フレーム数を返します。
3. `read()` でそれらを取得します。
4. 入力が終わったら `finish()` を一度呼びます。残りのテールが読み出し可能になります。
5. `reset()` でインスタンスをクリーンな状態に戻し、新しいストリームに使えます。

### 完全な使用例

```cpp
#include "pbshift/pbshift.h"
#include <vector>

void render(const std::vector<std::vector<float>>& in,   // [ch][frames]
            std::vector<std::vector<float>>& out,        // [ch][target]
            int sampleRate, double stretch, double pitchSemitones,
            bool preserveFormants) {
    const int ch = (int)in.size();
    const long long nIn = (long long)in[0].size();
    const long long target = (long long)llround(nIn * stretch);
    for (auto& c : out) c.assign(target, 0.0f);

    pbshift::Config cfg;
    cfg.sampleRate = sampleRate;
    cfg.channels   = ch;

    pbshift::Stretcher st;
    st.configure(cfg);
    st.setTimeStretch(stretch);
    st.setPitchSemitones(pitchSemitones);
    st.setFormantPreserve(preserveFormants);

    std::vector<const float*> ip(ch);
    std::vector<float*>       op(ch);
    long long fed = 0, got = 0;
    const int CHUNK = 8192;                        // どんなサイズでも OK
    while (fed < nIn) {
        const int k = (int)std::min<long long>(CHUNK, nIn - fed);
        for (int c = 0; c < ch; ++c) ip[c] = in[c].data() + fed;
        st.feed(ip.data(), k);
        fed += k;
        int avail = st.available();
        if (avail > 0 && got < target) {
            const int want = (int)std::min<long long>(avail, target - got);
            for (int c = 0; c < ch; ++c) op[c] = out[c].data() + got;
            got += st.read(op.data(), want);
        }
    }
    st.finish();                                   // テールをフラッシュ
    while (got < target && st.available() > 0) {
        const int want = (int)std::min<long long>(st.available(), target - got);
        for (int c = 0; c < ch; ++c) op[c] = out[c].data() + got;
        const int r = st.read(op.data(), want);
        if (r <= 0) break;
        got += r;
    }
}
```

### リアルタイムインサートでの使用

プラグイン／インサート文脈では、ホストブロックごとに `feed()` し、`available()` が返す分だけ `read()` してください。`inputLatency()` / `outputLatency()` をホストに報告すればプラグイン遅延補償（PDC）が機能します。`tests/rt_bench.cpp` がまさにこのパターンを実演し、ブロックごとの処理コストをリアルタイム予算と比較計測しています。

## CLI の使い方

リポジトリはオフラインのコマンドラインレンダラ（`tools/bin/pbshift[.exe]`）をビルドします。これはベンチマークハーネスのアダプタも兼ねています。

```
pbshift in.wav out.wav [--pitch <半音>] [--stretch <比率>] [--formant] [--voice] [--multi] [--tier live|offline]
```

| オプション | 意味 | 範囲 |
|---|---|---|
| `--pitch <st>` | ピッチシフト量（半音、小数可） | −24 … +24 |
| `--stretch <r>` | タイムストレッチ比 = 出力 / 入力の時間 | 0.25 … 4 |
| `--formant` | フォルマント保持を有効化 | フラグ |
| `--voice` | ボイスモード：音声/ボーカル向けの形状不変・倍音ロック位相 | フラグ |
| `--multi` | マルチモード：ドラム/ミックス向けの汎用トランジェント適応経路 | フラグ |
| `--tier` | レイテンシ/品質階層：`live`（約64ms）/ 既定（約128ms）/ `offline`（約256ms） | 列挙 |

例:

```sh
# 1 オクターブ上げ、フォルマント保持（音声向け）
pbshift vocal.wav vocal_up12.wav --pitch 12 --formant

# 半分の速度（2 倍の長さ）、ピッチはそのまま
pbshift drums.wav drums_2x.wav --stretch 2.0

# 同時指定: +7 半音 × 1.5 倍の長さ
pbshift mix.wav mix_p7_s15.wav --pitch 7 --stretch 1.5
```

入力: WAV（同梱のリーダ `tools/common/wav_io.h`）。出力: 入力と同じサンプルレートの 32-bit float WAV。

### オフライン・マルチ解像度レンダラ

マルチ解像度タイムストレッチエンジン（[上記](#マルチ解像度タイムストレッチオフライン)）はスタンドアロンのオフラインレンダラ `tools/bin/multires[.exe]` としても提供されます：

```
multires in.wav out.wav --stretch <ratio> [--voice] [--nopin] [--scales "N:lo:hi,..."]
```

| オプション | 意味 |
|---|---|
| `--stretch <r>` | タイムストレッチ比率 = 出力 / 入力時間 |
| `--voice` | 持続母音向けの単一長窓レイアウトを強制（未指定時はレイアウトを内容適応で自動選択） |
| `--nopin` | トランジェント・ピニングを無効化 |
| `--scales "N:lo:hi,…"` | 手動の帯域レイアウト: カンマ区切りの `fftSize:loHz:hiHz` スケール（実験用） |

```sh
# 内容適応マルチ解像度で 2× ストレッチ（打楽器・トーナル・音声を自動振り分け）
multires mix.wav mix_2x.wav --stretch 2.0
# 伸ばした歌唱母音を、単一長窓の音声レイアウトに強制
multires vowel.wav vowel_slow.wav --stretch 1.5 --voice
```

同じエンジンはライブラリ API からも、`Config::Tier::Offline` を設定しマルチ解像度経路を有効化することで利用できます。`Config::Mode::Voice` で音声レイアウトを選択します。ピッチシフトはストリーミング経路と同様に内部リサンプラを通ります。

## テストと品質ゲート

以下のゲートはすべて `tests/` の実行可能テストで強制されており、執筆時点で計測値とともにパスしています。

| ゲート | しきい値 | 計測値 |
|---|---|---|
| WOLA 恒等再構成（バイパスヌル） | < −120 dB | **−138 dB** |
| スペクトルピーク近傍の瞬時周波数精度（純音） | < 0.5 セント | **0.0002 セント** |

## ベンチマーク方法論

数値の再現に必要なものはすべて `benchmarks/` にあります。ハーネスはエンジン非依存です。共通の CLI 規約に従うエンジンなら何でも pbshift と並べてレンダリング・採点できます。比較したいエンジンがあればご自身でハーネスに掛けてください（本リポジトリが同梱するのは pbshift のみです）。

### 1. コーパス — `benchmarks/make_corpus.py`

タイムスケール変換評価の慣行に従った、完全に**決定論的**（固定シード）な 48 kHz float32 コーパス。歴史的にタイムスケール変換が苦手としてきたすべての信号クラスをカバーします。

| クラス | 信号 |
|---|---|
| 音声 | 実音声 TTS スピーチ（英語＋日本語。OS の音声合成 API でローカル生成 — 再現可能かつライセンスフリー）、ビブラート・グライド・/a/→/i/ フォルマントモーフ付きの合成歌唱母音 |
| パーカッシブ | 118 BPM ドラムループ（キック／スネア／ハイハット、グラウンドトゥルースのオンセット時刻付き）、スパースなカスタネット風クリック（古典的なトランジェント拷問テスト） |
| 撥弦／鍵盤 | ギターの分散和音＋ストラム、インハーモニシティとハンマーノイズ付きピアノコード |
| パッド／ミックス | デチューンされたステレオストリングパッド、フルステレオミックス（ベース＋ドラム＋アルペジオ） |
| 解析用 | 440 Hz サイン波、倍音豊富なトーン（A2）、対数スイープ、4 Hz AM トーン（変調スペクトル基準） |

パーカッション指標のため、グラウンドトゥルースのオンセット時刻を `corpus/ground_truth.json` に書き出します。

### 2. 条件

エンジン × 信号ごとに 14 条件:

- ストレッチのみ: **0.25×, 0.5×, 0.8×, 1.25×, 2.0×, 4.0×**
- ピッチのみ: **−24, −12, −5, +5, +12, +24 半音**
- 同時: **+7 半音 @ 1.5×** と **−7 半音 @ 0.75×**

ピッチシフトありの音声信号では、対応する全エンジンでフォルマント保持を有効にします。

### 3. 指標 — `benchmarks/metrics.py`

ストレッチ出力には時間対応した参照信号が存在しないため、**アーティファクト固有の量**を計測します。

| 指標 | 検出対象 |
|---|---|
| `duration_err` | 出力長 vs 期待値（ストレッチ × 入力） |
| `onset_recall` | オンセットの取りこぼし = トランジェントのなまり |
| `onset_precision` | 偽オンセット = 二重化／スタッタ |
| `attack_ratio` | 対応オンセットの出力/入力 10–90 % 立ち上がり時間（> 1 = なまり） |
| `hnr_db` | 倍音対倍音間ノイズ比（低下 = フェイジー／金属的ノイズ） |
| `f0_err_cents` | 出力 f0 の計測値 vs 期待値 |
| `envelope_lsd_db` | 時間対応フレーム上のスペクトル包絡の対数スペクトル距離（フォルマント品質） |
| `ltas_dist_db` | 長時間平均スペクトル距離（ストレッチのみの条件） |
| `warble_db` | 定常トーン上の 2–16 Hz の偽振幅変調 |
| `stereo_coh_drift` | 入力比のチャンネル間コヒーレンス損失 |

### 4. ランナー — `benchmarks/run_bench.py`

`tools/bin/` にあるすべてのエンジン実行ファイルにコーパス × 条件をレンダリングさせ、エンジンごとの中央値を集計して `benchmarks/out/results.csv` に書き出します。共通の CLI 規約を実装すれば、どんなエンジンでも参加できます。

```
engine in.wav out.wav --pitch <半音> --stretch <出力/入力比> [--formant]
```

### 再現手順

```sh

cd benchmarks
python make_corpus.py     # 決定論的コーパス -> corpus/*.wav
python run_bench.py       # tools/bin の全エンジン -> out/results.csv + サマリ

```

## 決定論の保証

pbshift は次を保証します: **同じ入力サンプル＋同じ設定 ⇒ ビット完全一致の出力。** 具体的には:

- **ホストバッファ非依存。** 64 サンプルずつ供給しても 8192 サンプルずつ供給しても、出力はバイト単位で同一です。バウンスとリアルタイム再生パスが一致します。
- **再レンダリングの同一性。** 同じ素材を 2 回レンダリングすればビットまで同一 — 複製して位相反転したトラックはデジタル無音までヌルアウトします。（`test_streaming` およびベンチマークマトリクスで検証済み。）
- **隠れたエントロピーなし。** すべての確率的要素はストリームの絶対位置からシードされ、実時間・メモリアドレス・スレッドタイミングには一切依存しません。

この性質はベストエフォートの挙動ではなく、CI の必須ゲートです — オフラインのバウンスとリアルタイムのパスが毎回デジタル無音にヌルします。

## レイテンシ

- レイテンシは設定に対して**固定**で、`inputLatency()` / `outputLatency()` がサンプル数で報告します。これをホストのプラグイン遅延補償（PDC)に渡してください。
- デフォルトの **Studio-RT** ティアは約 85 ms の解析窓（48 kHz で 4096 点 FFT、合成ホップ 1024 サンプル）を使い、48 kHz でおよそ 64–90 ms 帯の固定エンドツーエンドレイテンシを目標としています。
- 設計上のティア（`Config::Tier`）:

| ティア | 設計ポイント | 状況 |
|---|---|---|
| `Live` | 短い非対称窓、~40–58 ms | ロードマップ |
| `StudioRT` | ~64–90 ms 固定、約 1 コア | 現行デフォルト |
| `Offline` | 品質最優先、レイテンシ無制限 | 改良はロードマップ |

- `rt_bench` は任意のブロックサイズについて、平均／ワーストケースのブロックコストをリアルタイム予算と比較し、報告レイテンシとともに表示します。

## ロードマップ

- **マルチ解像度エンジンを既定へ。** 上記のオフライン・マルチ解像度タイムストレッチエンジンは実装済み・内容適応済みです。オプトインから既定の品質経路へ昇格させるかは、試聴（MUSHRA 方式）での確認を条件とします。
- **ストリーミング版マルチ解像度。** 現在のマルチ解像度経路は全信号（オフライン）です。ブロックストリーミング版により、コーラス感の無い品質をリアルタイムインサートにも提供します。
- **領域別の内容適応。** レイアウトは現在、信号ごとに一度だけ選択します。領域別（時間変化）検出により、1 つのトラック内でドラムフィルと持続パッドの間などでレイアウトを切り替えられます。
- **音声特化エンジン** — F0 トラッカ駆動の形状不変位相処理と有声／無声を考慮した位相制御により、極端なボーカルシフト（最大 ±24 半音）での自然さを最大化。
- **ノイズ成分モーフィング** — ノイズ／アンビエンス成分の専用再合成。大きなストレッチ比でもパッド・リバーブテール・テクスチャが金属的にならず自然に保たれます。
- **リアルタイム負荷の平準化** — 計算分割スケジューリングにより、ピッチシフト時のワーストケースのブロックコストを平坦化。
- **`Live` レイテンシティア** — 非対称の解析／合成窓ペアによる 60 ms 未満動作。
- **VST3 ラッパー** — ストリーミング API 上のリファレンス・リアルタイムインサートプラグイン。
- **ライセンスの確定** — 下記参照。
